# Class Reunion Web Application: Architectural Design

## 🏆 Executive Summary

The application is designed as a microservices architecture, running primarily in a serverless fashion (AWS Lambda) to achieve exceptional scalability and cost efficiency. The structure is separated into a highly responsive frontend (SPA) and secure, decoupled backend APIs.

## 📐 Technology Stack

| Component | Technology Chosen | Rationale |
| :--- | :--- | :--- |
| **Frontend (Client)** | **React (with TypeScript)** | Highly reactive, component-based architecture (SPA) ensures fast, smooth user experience. TypeScript adds vital type safety. |
| **Backend (API Logic)** | **Node.js (TypeScript) / AWS Lambda** | Extremely fast for I/O-bound operations (API calls/DB queries), minimizing cold start times. Excellent performance within the serverless environment. |
| **API Layer** | **Amazon API Gateway** | Provides a managed, scalable endpoint for the Lambda functions, handling authentication, rate limiting, and routing. |
| **Database** | **AWS Aurora Serverless (PostgreSQL)** | Best balance of cost, searchability, and integrity. Relational structure simplifies complex queries required for list/search views, while Serverless minimizes idle costs. |
| **File Storage** | **Amazon S3** | Standard for scalable, durable, and low-cost object storage. Ideal for high-volume photo uploads. |
| **Deployment/DevOps** | **Serverless Framework (AWS SAM)** | Simplifies defining, deploying, and managing the entire serverless stack (API Gateway, Lambda, IAM roles). Crucial for local development simulations. |
| **Local Development** | **Docker / Localstack** | Allows for simulating the entire AWS ecosystem (Lambda runtime, S3, Aurora) locally, enabling a fully integrated development workflow. |

## 🗺️ High-Level Architecture Diagram (Conceptual Flow)

The application utilizes a decoupled microservices pattern:

1.  **User interacts with Frontend (React SPA).**
2.  **Frontend sends requests to AWS API Gateway.**
3.  **API Gateway authenticates (Keyword Validation) and routes requests to specific Lambda functions.**
4.  **Lambda Functions interact with:**
    *   **Aurora Serverless (Read/Write)** for profile/comment data.
    *   **S3 (Read/Write)** for photos (Photos are uploaded directly by the client to a secure pre-signed S3 URL generated by the Lambda).
5.  **Data Flow:** Photos never pass through the Lambda logic; only metadata pointers are saved to the database.

## 💾 Data Model Design (Schema)

The application is built around five primary entities:

### 📚 1. School (Relational)
*   `school_id` (PK)
*   `name` (Public)
*   `location` (Public)

### 🎓 2. Class (Relational)
*   `class_id` (PK)
*   `school_id` (FK to School)
*   `year` (e.g., 2005, 2025) (Public)

### 👤 3. User (Relational)
*   `user_id` (PK)
*   `school_id` (FK to School)
*   `class_id` (FK to Class)
*   `email` (Unique, Used for Keyword Login)
*   `keyword` (Secure Hashed Field)
*   `is_admin` (Boolean)
*   `first_name` (Public)
*   `last_name` (Public)
*   `nickname_school` (Public)
*   `phone` (Private)
*   `bio` (Private/Configurable)
*   `address` (Private/Configurable)
*   `then_photo_url` (S3 URL)
*   `now_photo_url` (S3 URL)
*   `is_first_name_public` (Boolean) *(Controls field visibility)*

### 💬 4. Comment (Relational)
*   `comment_id` (PK)
*   `target_user_id` (FK to User, the person being commented on)
*   `commenter_id` (FK to User, who wrote the comment)
*   `content` (Text, max 1 page)
*   `is_published` (Boolean, defaults to `false`)
*   `created_at`

## ✅ Key Design Decisions

*   **Keyword Login:** Authentication relies on email validation against a pre-seeded, securely hashed keyword field.
*   **Data Privacy:** Non-name fields default to private; visibility is controlled by a flag.
*   **Scalability:** Serverless architecture ensures automatic scaling and low operational costs.
*   **Cost Control:** Aurora Serverless minimizes expenditure on idle database time.
*   **Photo Uploads:** S3 Pre-signed URLs prevent high-volume media from overloading the Lambda function and API Gateway.
