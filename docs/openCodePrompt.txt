🏆 Class Reunion: Architectural Design Document
1. Executive Summary
The application is designed as a microservices architecture, running primarily in a serverless fashion (AWS Lambda) to achieve exceptional scalability and cost efficiency. The structure is separated into a highly responsive frontend (SPA) and secure, decoupled backend APIs.
2. Technology Stack Selection
Component	Technology Chosen	Rationale
Frontend (Client)	React (with TypeScript)	Highly reactive, component-based architecture (SPA) ensures fast, smooth user experience. TypeScript adds vital type safety, improving long-term maintainability.
Backend (API Logic)	Node.js (TypeScript) / AWS Lambda	Node.js is extremely fast for I/O-bound operations (like API calls/DB queries), minimizing cold start times on Lambda. It provides excellent performance within the serverless environment.
API Layer	Amazon API Gateway	Provides a managed, scalable endpoint for the Lambda functions, handling authentication, rate limiting, and routing to the backend.
Database	AWS Aurora Serverless (PostgreSQL)	Best balance of cost, searchability, and integrity. Unlike DynamoDB, relational structure (School -> Class -> User) is natively handled by SQL, which simplifies complex queries needed for list/search views. Aurora Serverless scales down to near-zero cost when idle.
File Storage	Amazon S3	The industry standard for scalable, durable, and low-cost object storage. Ideal for high-volume photo uploads.
Deployment/DevOps	Serverless Framework (AWS SAM)	Simplifies defining, deploying, and managing the entire serverless stack (API Gateway, Lambda, IAM roles). Crucial for local development simulations.
Local Development	Docker / Localstack	Use Docker to simulate the entire AWS ecosystem (Lambda runtime, S3, DynamoDB/Aurora) locally, allowing for a fully integrated development workflow without requiring constant AWS access.
3. High-Level Architecture Diagram (Conceptual Flow)
The application follows a standard decoupled microservices pattern:
1. User interacts with Frontend (React SPA).
2. Frontend sends requests to AWS API Gateway.
3. API Gateway authenticates (Keyword Validation) and routes requests to specific Lambda functions.
4. Lambda Functions interact with:
- Aurora Serverless (Read/Write) for profile/comment data.
- S3 (Read/Write) for photos (Photos are uploaded directly by the client to a secure pre-signed S3 URL generated by the Lambda).
5. Data flows are secure: Photos never pass through the Lambda logic, only metadata pointers are saved to the database.

We need five primary entities:

5. Key Design Decisions & Implementation Notes
A. Authentication (Keyword Login)
- Mechanism: When a user attempts to log in, the client sends the email and the keyword.
- Lambda Logic: The authentication Lambda retrieves the user's record from the database and compares the input keyword against the securely hashed keyword stored in the User table.
- Security: The keyword should be hashed using a robust, slow algorithm (e.g., Argon2 or scrypt) to prevent dictionary attacks, even if it's not a "password."
- Initial Access: The system requires a master list of graduates (the "class list") to be pre-seeded into the database via the Admin role.
B. Data Privacy (Public/Private Fields)
- All fields (phone, bio, address) default to private.
- The profile page must conditionally render fields based on the user's privacy preferences, which are saved in the User entity. The public API should only serve data explicitly marked as public.
C. Comments Constraint (Third Person)
- This is a UX/Content Management System (CMS) constraint, not a database constraint.
- Implementation: The comment submission form on the frontend should include instructions/guidance, and the backend should validate the comment text against a basic regex pattern checking for third-person pronoun usage if a strict policy is desired.
D. Scalability & Performance
- Lambda + API Gateway: Provides automatic scaling to handle massive spikes in traffic (e.g., reunion day), ensuring low latency.
- Aurora Serverless: Pay-per-use model avoids paying for idle database resources, meeting the low-cost requirement.
- S3 Direct Upload: By generating pre-signed URLs, the photo upload load is entirely taken off the Lambda function, keeping the API lean and extremely fast.
This architecture delivers a highly resilient, scalable, and cost-optimized solution that meets all of your core requirements.
